O que é teste de Software?

Teste de Software é um processo do desenvolvimento de software, que tem como objetivo encontrar falhas e bugs. A idéia é verificar se o software entrega tudo aquilo que ele propõe. Testar um software implica em qualidade de software, pois é através do teste que validamos as funcionalidades que ele propõe.

Como testar um software?

Existem várias técnicas para testar um software e através delas o software é adequadamente testado. O software tem que ser submetido a um teste em três momentos: durante o desenvolvimento, após o desenvolvimento e na homologação, ou seja, no momento em que o software será ativado para ser utilizado.

Testar o software e não encontrar erros significa que o sofware não possui falhas?

Através da atividade de teste pode-se detectar apresença de erros em um programa, embora não se possa concluir a ausência deles.

Erro, defeito e falha

Uma pessoa pode cometer um erro (engano), que pode levar à introdução de um defeito (falha ou bug) no código do software ou em algum outro produto de trabalho relacionado. Um erro que leva à introdução de um defeito em um produto de trabalho pode acionar outro erro que leva à introdução de um defeito em um outro produto de trabalho relacionado. Por exemplo, um erro de elicitação de requisitos pode levar a um defeito de requisitos, o que resulta em um erro de desenvolvimento que leva a um defeito no código.

Erro

Erro humano produzindo resultado incorreto. O erro evidencia o defeito, ou seja, quando há diferença entre o valor obtido e o valor esperado constitui um erro.

Defeito

É uma inconsistência no software, algo que foi implementado de maneira incorreta. Ele ocorre em uma linha de código, como uma instrução errada ou um comando incorreto. O defeito é a causa de um erro, porém se uma linha de código que contém o defeito nunca executar, o defeito não vai provocar um erro.

Falha

É um comportamento inesperado do software. Uma falha pode ter sido causada por diversos erros, mas alguns erros podem nunca causar uma falha. Exemplo prático: Houve um retorno de um valor não esperado, como null, isso é um erro, e por causa desse null ocasionou uma falha no sistema.

Casos de Teste e Cenários de Teste

Caso de teste é um conjunto de ações executadas para verificar um determinado recurso ou funcionalidade do aplicativo.
Cenário de teste é qualquer funcionalidade do aplicativo que pode ser testada.

Niveis de Teste

De acordo com ISTQB® (International Software Testing Qualifications Board), os níveis de teste são grupos de atividades organizadas e geridas em conjunto. Cada nível é uma instância do processo de teste e estão divididos da seguinte forma:

  1. Teste de Componente ou Unitário: Testa unidades individuais de programa de forma independente. Um exemplo uma função de soma entre dois números, o resultado da soma de 1 e 2 deve sempre ser 3.O teste de componente geralmente é realizado isoladamente do resto do sistema
  2. Teste de Integração: É o teste entre diferentes módulos em um sistema. Um exemplo para esse teste de requisições HTTP, o mais comum para servidores.
  3. Teste de Sistema: O sistema é testado como um todo. Testa uma release de sistema que será entregue ao cliente.
  4. Teste de Aceite: é a ação de teste final antes da implementação do software. A meta do teste de aceitação é verificar se o software está pronto e pode ser utilizado pelos usuários, para desempenhar as funções e tarefas para as quais o software foi construído.

Tipos de Teste

  1. Teste funcional: Essa técnica testa os requisitos funcionais, as funções e os casos de uso. Exemplo: Realizar o teste de aceitação do usuário, para avaliar se o software faz o que promete.
  2. Teste não funcional: Avaliam as características de sistemas e de softwares, como usabilidade, eficiência de performance ou segurança
  3. Teste relacionado a mundaça: testar para confirmar se as alterações corrigiram o defeito ou implementaram a funcionalidade corretamente e não causaram consequências adversas imprevistas.
  4. Teste de caixa-branca: é derivado de testes com base na estrutura interna ou na implementação do sistema. A estrutura interna pode incluir código, arquitetura, fluxos de trabalho e fluxos de dadosdentro do sistema.
  5. Teste exploratório: É um técnica que tem o foco em explorar (vasculhar) um software para entender o que ele faz, o que não faz, o que funciona e o que não funciona.
  6. Teste de segurança: Permite avaliar as vulnerabilidades em aplicações serviços frente a diferentes tipos de ataques de segurança.
  7. Teste de Performance: Testar o software a fim de encontrar o seu limite de processamento de dados no seu melhor desempenho.

Outros conceitos de Teste

  1. Teste stress: Testes de estresse são usados para garantir a estabilidade e a confiabilidade do sistema. Os testes simulam uma alta taxa de dados e de tráfego e registram como o sistema responde nestas condições.
  2. Teste de Carga: É a prática de simular o uso do mundo real, ou carregar, em qualquer software, site, aplicativo web, API ou sistema para analisar e identificar fatores como capacidade de resposta, degradação e escalabilidade.
  3. Teste de Conformidade: É uma técnica de teste não funcional para determinar se um sistema atende aos padrões prescritos.
  4. Teste de Usabilidade: Os testes são realizados com usuários representativos do público-alvo. Cada participante tenta realizar tarefas típicas enquanto o analista observa, ouve e anota.

Teste de caixa-preta na prática

Nos testes de caixa-preta, o foco é apenas as entradas e saídas do sistema, sem se preocupar com a estrutura interna do programa.

Técnicas de teste caixa-preta

Particionamento de equivalência

O particionamento de equivalência divide os dados em partições (também conhecidas como classes de equivalência), de tal forma que todos os membros de uma determinada partição deve ser processado da mesma maneira (ver Kaner (2013) e Jorgensen (2014)). Existem partições de equivalência para valores válidos e inválidos.

Análise de valor limite

A análise de valor limite é uma extensão do particionamento de equivalência, mas só pode ser usada quando a partição é ordenada, consistindo em dados numéricos ou sequenciais. Os valores mínimo e máximo (ou primeiro e último valores) de uma partição são seus valores limites (Beizer, 1990).

Teste de tabela de decisão

Técnicas de teste combinatórias são úteis para testar a implementação de requisitos do sistema que especificam como diferentes condições de combinações levam à resultados diferentes. Uma abordagem para esse teste é o teste de tabela de decisão.

Profissionais de Teste

  1. Testers: Essa técnica testa os requisitos funcionais, as funções e os casos de uso. Exemplo: Realizar o teste de aceitação do usuário, para avaliar se o software faz o que promete.
  2. Analista de Testes: É uma técnica do teste de software que consiste na aplicação de versões mais recente do software, para garantir que não surgiram novos defeitos em componentes já analisados.
  3. Gerente de Testes: Uma das principais atividades de um Gerente de Teste é garantir e utilizar recursos que tem disponíveis (pessoas, software, hardware, infraestrutura etc.) para a realização de processos que agreguem valor.
